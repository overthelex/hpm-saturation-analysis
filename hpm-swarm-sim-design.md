# Симуляция «рой vs эшелонированный HPM-рубеж» — проект

**Оборонный инструмент поиска порогов насыщения. Дата: 2026-07-20.**

> **Назначение и рамки.** Симуляция — это **инструмент оценки обороны**: она прогоняет
> абстрактные манёвры роя против конфигурации рубежа и выдаёт долю прорыва (leak-rate),
> цену атаки и карту слабых мест. Выход — «где и почему рубеж рвётся», чтобы оборона
> знала, что чинить. Это НЕ планировщик атаки под конкретный реальный объект: кинематика
> абстрактная, координаты условные, цель — пороговая аналитика, а не боевое наведение.
> Модель, которую симуляция уточняет, — `hpm-saturation-model.md`.

---

## 1. Зачем симуляция, а не только формулы

Замкнутые формулы `hpm-saturation-model.md` (§5, §7) точны для **одной апертуры**. Но
englobing-атака в тыл (§8) зависит от вещей, которые в закрытом виде не берутся:

- fratricide-дуги нескольких апертур пересекаются нетривиально в 3D;
- назначение целей (target assignment) — это контенция за общий ресурс лучей;
- подвижные апертуры → швы движутся во времени;
- многовысотная атака бьёт по зенитной дыре и по возвышению стыков одновременно;
- «доля_шва» из §8.3 зависит от качества ISR роя (полное знание vs слепая атака).

Всё это — динамика с дискретными событиями и геометрическими ограничениями. Правильный
инструмент — **агентная симуляция с шагом по времени в 3D**, а не Монте-Карло по формуле.

---

## 2. Рекомендуемый метод (главный ответ на «как лучше»)

**3D кинематическая time-stepped агентная модель на Python + NumPy, векторизованная по
дронам, с обёрткой Монте-Карло для развёрток.** Обоснование выбора по осям:

| Ось выбора | Решение | Почему |
|---|---|---|
| Парадигма | Агентная + дискретный шаг времени | Fratricide-дуги, назначение целей, движение апертур не берутся аналитически |
| Физика | Кинематика (точки + векторы скорости), без аэродинамики | Нужны пороги геометрии, не лётные характеристики; на порядок дешевле |
| Шаг Δt | 10–50 мс | При v≤50 м/с это ≤2.5 м/шаг; t_c=0.1–1 с разрешается на 2–100 шагов |
| Язык | Python 3 + NumPy | N~1000 дронов векторизуются per-tick; M апертур — обычный цикл (M мало); лёгкие развёртки |
| Масштаб | Vectorize по дронам, seed-детерминизм | 1 прогон N=1000, T=60 с, Δt=20 мс ≈ 3000 тиков — доли секунды; тысячи прогонов Монте-Карло — минуты |
| Ускорение (если нужно) | Numba/JAX на «горячем» ядре, либо Rust только при >10⁶ прогонов | Не усложнять раньше времени; сначала профилировать |
| Оптимальность обороны | Жадный EDF по умолчанию + опция Hungarian для верхней границы | Жадный ≈ реальный fire-control; оптимальный назначает границу «лучшей возможной обороны» |

**Не рекомендую:** физдвижки (Gazebo/Unity) — избыточны, лётная точность не нужна;
чистый Монте-Карло по формуле — не ловит fratricide/движение; ML/RL-рой — это уже
проектирование атаки, вне рамок (§ Границы).

## 3. Архитектура (разделение ответственности)

```
World            — 3D сцена, время, Δt, детерминированный RNG(seed)
├─ Asset         — защищаемый объект: позиция, keep-out радиус ρ
├─ Defense
│  ├─ Aperture[] — позиция (моб.: траектория), θ (ширина луча), el_max,
│  │               slew-модель (ω град/с), t_c, n_cone, P_k(r), hardened?
│  ├─ NoFireMask — предвычисленные запретные дуги: asset keep-out ∪ соседи ∪ (±margin)
│  ├─ Sensor     — дальность обнаружения, T_max треков, задержка захвата
│  └─ Scheduler  — назначение цель→апертура (EDF greedy | Hungarian)
├─ InnerLayer    — опц. кинетический пузырь: радиус, темп, magazine depth, P_k
└─ Threat
   ├─ Drone[]    — позиция, v-вектор, alive, жив/сбит/прорвался
   ├─ Spawner    — старт на сфере/оболочке; распределение по Ω и высотам
   └─ Guidance   — политика наведения (см. §5); уровень ISR роя
Logger/Metrics   — покадровый лог + агрегаты; Sweep-драйвер поверх
```

**Ключевой инвариант fratricide (сердце модели):** на каждом тике дрон `d` **поражаем**
апертурой `a` только если он в конусе `a`, в пределах `R_eff`, в пределах `el_max`,
**и не** в `NoFireMask[a]`. Если `d` вне легальной дуги **всех** апертур → он в слепом
объёме → гарантированный прорыватель (пока не входит в `InnerLayer`).

## 4. Цикл fire-control (на каждом тике)

```
1. Sensor: обновить треки (обнаружение при r<R_det, до T_max одновременно)
2. Для каждой апертуры: список легальных целей = в конусе ∧ r<R_eff ∧ el<el_max ∧ ∉NoFireMask
3. Scheduler: назначить цели апертурам
   - greedy EDF: каждой свободной апертуре — легальная цель с минимальным
     временем-до-контакта τ_d=(r_d−R_c)/v_d (самая срочная); учесть время slew
   - опц. Hungarian: глобально-оптимальное назначение (верхняя граница обороны)
4. Апертуры доводят луч (slew), по достижении — импульс: гасят до n_cone целей в конусе
   с вероятностью P_k(r); занятость на t_c
5. InnerLayer: цели в кинетич. радиусе гасятся по темпу/magazine
6. Kinematics: сдвинуть живых дронов на v·Δt; апертуры — по траектории (если мобильны)
7. Учёт: r<R_c ∧ alive → LEAKER; alive ∧ вне поля → escaped
```

## 5. Манёвры роя (сценарии-входы, абстрактно)

Моделируем как **политики наведения**, параметризованные уровнем ISR — от слепого до
идеального знания конфигурации. Именно контраст между ними даёт «цену знания швов».

| # | Манёвр | Что проверяет | Уровень ISR |
|---|---|---|---|
| S0 | Равномерная оболочка, синхронно | Базовая угловая насыщенность §5 | нулевой |
| S1 | Секторно-синхронная волна | Насыщение конкретных швов | средний |
| S2 | Многовысотный englobing | Зенитная дыра + возвышение стыков §8.3 | средний |
| S3 | Прорыв по швам с полным знанием конфигурации | Верхняя оценка «доля_шва→1» (§8.3) | идеальный |
| S4 | Тыловой заход после пересечения кольца | Внутренний слепой объём §8.2 | идеальный |
| S5 | S3/S4 против **подвижных** апертур | Обесценивание ISR мобильностью (замок 2) | устаревающий |

Каждый сценарий — распределение стартов по `Ω_def` и высотам + правило перенацеливания.
Идеальный ISR = рой знает `NoFireMask(t)`; устаревающий = знает с задержкой Δ_isr, апертуры
за это время переехали.

## 6. Развёртки и метрики (те же три, что в модели)

**Оси развёртки (Монте-Карло, сетка ×N сидов):**
конфигурация обороны `M`, `R_ring`, `θ`, `el_max`, `t_c`, `n_cone`, подвижность,
экранирование своих; параметры атаки `N`, `v`, высотный разброс, ISR-уровень, сценарий.

**Метрики (в согласии с `hpm-saturation-model.md` §6):**
- **N_min** — минимальное `N`, при котором `P(leak≥1) > 0` (бинпоиск по `N` на сценарий).
- **leak-rate** — доля прорвавшихся vs `N` и конфигурация; кривая с плато.
- **Цена атаки** — сбито до первого прорыва; привязка к стоимости планера.
- **Диагностика:** доля прорывов через зенит / через шов / в тыл; сколько тактов апертуры
  простаивали из-за `NoFireMask` (fratricide-forced holds) — прямой измеритель §8.

**Артефакты:** heatmap `leak(M × экранирование)`, `leak(высотный разброс × el_max)`,
`N_min(θ × t_c)` (кросс-чек с таблицей §7), `цена_знания = leak(S3) − leak(S0)`.

## 7. Валидация (обязательна до доверия)

Симуляция должна **воспроизвести аналитику**, прежде чем ей верить на сложных случаях:

1. **Вырожденный случай = 1 апертура, S0:** leak-порог обязан совпасть с `T_r/τ` из §5/§7.
   Если сим и формула расходятся — баг в планировщике или геометрии.
2. **Предельные случаи:** `v→0 ⇒ leak→0`; `θ→90° ⇒ S→1` (нет угловой насыщенности);
   `n_cone→∞ ⇒ сервисный предел снят, остаётся только угловой`.
3. **Fratricide off (экранирование=идеальное):** внутренний слепой объём исчезает,
   `leak_тыл→0` — проверка, что маска подключена правильно.
4. **Детерминизм:** один seed → бит-в-бит один результат (иначе развёртки невоспроизводимы).
5. **Сходимость Монте-Карло:** leak-оценка стабильна по числу сидов (доверит. интервал).

## 8. План реализации (поэтапно)

| Этап | Содержание | Проверка |
|---|---|---|
| 0 | Скелет: World, Δt-цикл, кинематика дронов, 1 статич. апертура, S0 | воспроизводит `T_r/τ` §7 |
| 1 | P_k(r), n_cone, сенсор/T_max, greedy-EDF планировщик | leak-кривая гладкая, пределы §7.2 |
| 2 | Кольцо `M` апертур + `NoFireMask` (fratricide/keep-out/зенит) | fratricide-off ⇒ leak_тыл=0 |
| 3 | Сценарии S1–S4, многовысотность, уровни ISR | S3>S0 (цена знания швов >0) |
| 4 | Мобильные апертуры (траектории) + S5, InnerLayer (кинетика) | S5 ≪ S4 (мобильность закрывает тыл) |
| 5 | Sweep-драйвер, Монте-Карло, heatmap-артефакты | сходимость §7.5 |
| 6 | (опц.) Hungarian-планировщик, Numba/JAX если профиль требует | Hungarian ≥ greedy (верхняя граница) |

**Технологии:** Python 3.11+, NumPy; `scipy.optimize.linear_sum_assignment` (Hungarian);
`matplotlib` для heatmap; `pytest` для валидации §7; конфиги — dataclass/YAML; сиды —
явные, без глобального RNG. Структура: `sim/` (движок), `scenarios/` (S0–S5),
`configs/` (обороны), `analysis/` (развёртки/графики), `tests/` (валидация).

## 9. Что симуляция НЕ делает (границы)

- Не оптимизирует атаку под реальный объект: координаты условны, наведение — абстрактные
  политики, не боевой автопилот. Выход — пороги обороны, не маршруты прорыва.
- Не моделирует лётную динамику, РЭБ-детали, тепловые переходные процессы отдельных LRAM —
  только их огрублённые эффекты через параметры (`t_c`, duty, P_k). Все упрощения
  **ужесточают** оборонный предел, т.е. реальные пороги не выше расчётных.
- Не заменяет модель, а уточняет её на многоапертурных/динамических случаях, где §5–7
  замкнутого решения не имеют.
